## Database Engine

Here is the exact content to fill the `database_engine` section in `DATABASE_SCHEMA.md`:

**Engine:** SQLite 3.45+ with WAL (Write-Ahead Logging) mode
**Deployment:** Single file at `data/leylcafe.db`; persisted via Docker volume mount in both dev and production (Azure Files volume on Container Apps).

### Development Connection String

```
sqlite+aiosqlite:///./data/leylcafe_dev.db
```

Docker Compose: file is bind-mounted from `./data/` on the host:

```yaml
volumes:
  - ./data:/app/data
```

Environment variable reference (`.env`):

```
DATABASE_URL=sqlite+aiosqlite:///./data/leylcafe_dev.db
```

### SQLite Configuration (applied at connection startup)

```sql
PRAGMA journal_mode=WAL;      -- enable WAL for concurrent reads + writes
PRAGMA foreign_keys=ON;       -- enforce FK constraints (off by default in SQLite)
PRAGMA synchronous=NORMAL;    -- safe + performant for WAL mode
PRAGMA busy_timeout=5000;     -- 5 s wait before SQLITE_BUSY error
```

Applied via SQLAlchemy event listener on engine `connect` event; no manual `PRAGMA` calls needed in application code.

### Note on UUIDs and Types

SQLite has no native UUID or TIMESTAMPTZ type. The schema uses:
- **TEXT** for UUID primary keys — generated by Python (`str(uuid.uuid4())`)
- **DATETIME** for timestamps — stored as ISO-8601 strings (`2026-03-04T14:00:00`)
- **NUMERIC** / **REAL** for monetary amounts (SQLite NUMERIC affinity)
- **INTEGER** for booleans (0 / 1)

## Indexes & Performance

| Table | Columns | Type | Reason |
|-------|---------|------|--------|
| users | email | UNIQUE B-tree | Login lookup; enforce no duplicate accounts |
| categories | (name, parent_id) | UNIQUE B-tree | Prevent duplicate category names within the same parent |
| products | category_id | B-tree | FK; filter/join products by category |
| products | (category_id, is_active) | Composite B-tree | Filtered product listing: active products per category |
| products | sku | UNIQUE B-tree (partial: WHERE sku IS NOT NULL) | Enforce SKU uniqueness only when assigned |
| recipe_items | product_id | B-tree | FK; fetch recipe for a product (COGS calculation) |
| recipe_items | ingredient_id | B-tree | FK; find all products that use an ingredient |
| recipe_items | (product_id, ingredient_id) | UNIQUE Composite | Prevent duplicate ingredient rows per product recipe |
| ingredients | current_stock | B-tree (partial: WHERE current_stock <= reorder_level) | Low-stock alert query; dashboard badge count |
| stock_entries | ingredient_id | B-tree | FK; all entries for a given ingredient |
| stock_entries | (ingredient_id, entered_at DESC) | Composite B-tree | Stock valuation and history timeline ordered by date |
| stock_entries | created_by | B-tree | FK; audit trail per user |
| stock_movements | ingredient_id | B-tree | FK; deduction history per ingredient |
| stock_movements | (ingredient_id, created_at DESC) | Composite B-tree | Running-stock queries ordered by time |
| stock_movements | reference_id | B-tree | Reverse-lookup: find movements caused by a sale_item or stock_entry |
| sales | sale_date | B-tree | Date-range scans for daily/weekly/monthly reports |
| sales | (sale_date, cashier_id) | Composite B-tree | Filter sales by date range and user simultaneously |
| sales | cashier_id | B-tree | FK; sales per staff member |
| sale_items | sale_id | B-tree | FK; all line items for a sale (most frequent join) |
| sale_items | product_id | B-tree | FK; aggregate sales per product across all sales |
| sale_items | (product_id, sale_id) | Composite B-tree | Product revenue roll-up joined to sales.sale_date |
| expenses | expense_date | B-tree | Date-range scans for P&L reports |
| expenses | (expense_date, expense_category_id) | Composite B-tree | Expense breakdown by category within a period |
| expenses | expense_category_id | B-tree | FK; group expenses by category |
| expenses | created_by | B-tree | FK; audit trail per user |
| token_denylist | jti | UNIQUE B-tree | O(log n) revocation check on every authenticated request |
| token_denylist | expires_at | B-tree | Periodic TTL cleanup: DELETE WHERE expires_at < NOW() |
| file_imports | created_by | B-tree | FK; import history per user |
| file_imports | (status, created_at DESC) | Composite B-tree | Queue polling: pending imports ordered by submission time |

---

**Design decisions:**
- Every FK column has a dedicated index; SQLite does not create FK indexes automatically.
- Composite indexes follow selectivity order: high-cardinality column first (date, ingredient_id) so the planner can use the index for single-column predicates as well.
- Partial indexes (`WHERE sku IS NOT NULL`, `WHERE current_stock <= reorder_level`) eliminate dead rows from the index, keeping it small and the low-stock alert query near-instant even as the ingredients table grows.
- `token_denylist.jti` is UNIQUE to double-enforce one-revocation-per-token at the database layer, not just the application layer.
- `(status, created_at DESC)` on `file_imports` covers the import worker's queue poll (`WHERE status = 'pending' ORDER BY created_at ASC LIMIT 1`) with a single index scan.

### File Naming Convention

```
{NNNN}_{snake_case_description}.py
```

| Segment | Rule |
|---|---|
| `NNNN` | Zero-padded 4-digit sequence starting at `0001`. Monotonically increasing; never reuse. |
| `snake_case_description` | Imperative verb phrase, max 6 words. Examples: `add_users_table`, `add_stock_movement_index`, `drop_legacy_notes_column`. |

Generated revision IDs (Alembic default hash) are retained inside the file but are **not** part of the filename. The numeric prefix guarantees deterministic ordering in `alembic history` output and in version-control diffs.

### Running Migrations

**Prerequisites:** `DATABASE_URL` environment variable must be set before running any Alembic command.

```bash
# Apply all pending migrations (up)
alembic upgrade head

# Roll back the last applied migration (down)
alembic downgrade -1

# Roll back to a specific revision
alembic downgrade 0003_add_expense_categories

# Show current applied revision
alembic current

# Show full migration history
alembic history --verbose

# Generate a new autogenerated revision after editing models
alembic revision --autogenerate -m "add_stock_movement_index"
```

Every migration file **must** implement both `upgrade()` and `downgrade()`. A `downgrade()` that raises `NotImplementedError` is rejected in code review. This policy enables safe rollback during a failed deployment slot swap.

In CI/CD (GitHub Actions), `alembic upgrade head` runs as the first step of the deploy job, before the new container revision receives traffic. If the migration fails, the pipeline halts and the previous revision continues serving requests.

### Seeding Strategy for Development

Seeding is implemented as a standalone Python script, not as an Alembic migration, to prevent seed data from polluting the production migration chain.

```
backend/
  scripts/
    seed_dev.py     # inserts reference + fixture data via SQLAlchemy session
```

**Execution:**
```bash
# From repo root, with dev DATABASE_URL pointing to local postgres
python -m scripts.seed_dev
```

**Seed data included:**

| Table | Seed content |
|---|---|
| `users` | 1 owner (`owner@leyl.dev`), 1 manager (`manager@leyl.dev`), bcrypt-hashed passwords `dev1234` |
| `categories` | 6 rows: Sıcak İçecek, Soğuk İçecek, Tatlı, Tuzlu, Kahve, Diğer |
| `products` | 20 rows covering all categories, with `unit_cost` and `sale_price` |
| `stock_movements` | 30 rows of `IN` entries (initial stock) spread across last 14 days |
| `sales` + `sale_items` | 90 days of synthetic daily sales (≈15–30 items/day) for chart rendering |
| `expenses` | 90 days of recurring expense entries (rent, electricity, wages, supplies) |

The seed script is **idempotent**: it checks for existing rows by natural key before inserting. Running it twice produces no duplicates.

Seed data is **never** run in production. The deploy pipeline checks `APP_ENV != development` and skips `seed_dev.py` unconditionally.

---

**Key decisions embedded in this strategy:**
- Tool is Alembic with aiosqlite (locked by ADR-002 — FastAPI + SQLAlchemy stack; SQLite dialect)
- `downgrade()` is mandatory on every revision to support zero-downtime slot-swap rollback
- `alembic upgrade head` is a deploy-gate step in the GitHub Actions pipeline
- Seed data lives outside the migration chain to keep production migrations clean
